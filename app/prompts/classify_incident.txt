# FRAUD DETECTION RULES

**PRIORITY #1:** Check if medical certificate contradicts the claim (healthy patient claiming illness)

## 1. Contradictory Evidence
Medical documentation contradicts claim description.

**CRITICAL:** Compare what the claim says vs what the medical certificate actually states.

**Fraud pattern:**
- Claim says: "unable to travel/work/participate due to medical condition"
- Certificate says: patient is HEALTHY, FIT, or has NO restriction/contraindication to activity

**How to detect:**
1. Read the claim description - what does claimant say happened?
2. **CHECK EXTRACTED FIELDS FIRST:** Look at `medical_finding` and `diagnosis` fields in document summaries - these show what was actually CHECKED/selected on the form
3. **Understand negations (CRITICAL):**
   - Certificate states "NO" + [contraindication/restriction/illness] = patient is HEALTHY/FIT
   - Certificate states patient "HAS NO" medical issue = patient is HEALTHY
   - Certificate states patient "CLEARED/FIT/ABLE" for activity = patient is HEALTHY
   - Certificate permits/authorizes activity (including competition) = patient is HEALTHY
4. **Distinguish fitness vs restriction (these are OPPOSITES):**
   - "No contraindication to [activity]" = FITNESS certificate = patient CAN do activity = HEALTHY
   - "Must be exempt from [activity]" = RESTRICTION = patient CANNOT do activity = SICK
   - Don't confuse these - they have opposite meanings!
5. **CHECKBOX FORMS:** Medical certificates often list multiple options - only trust what's in the `medical_finding` field (that's what was CHECKED)
6. Compare the meaning: Does the certificate indicate health/fitness while the claim says illness/inability?
7. **If contradiction found:** This is FRAUD - healthy patient falsely claiming medical emergency

**Examples of contradictions:**
- Claim: "couldn't participate in sports camp" | Certificate indicates: patient cleared for sports/competition
- Claim: "too sick to work" | Certificate indicates: patient fit to return to work
- Claim: "unable to travel due to illness" | Certificate indicates: patient healthy with no restrictions

**Common scenarios:**
- Sports camp cancellation: claimant says "couldn't participate due to medical issue" BUT certificate states patient cleared/fit for sports → FRAUD
- Work absence: claimant says "too sick to work" BUT certificate states patient cleared to return to work → FRAUD
- Travel cancellation: claimant says "unable to travel due to illness" BUT certificate states patient healthy with no restrictions → FRAUD

**CRITICAL - When contradiction detected:**
- **MUST set:** `is_healthy_patient_certificate=true`
- **MUST set:** `covered_reason=false`
- **MUST set:** `confidence=0.90`
- **MUST explain in reasoning:** "Certificate states [what it says about health/fitness] but claimant says [what they claimed]. This is contradictory evidence."
- This triggers automatic DENIAL for fraudulent claims

**Before finalizing your response:** Ask yourself: "Does the medical certificate actually show an ILLNESS/INJURY that prevented the activity, or does it show the patient is HEALTHY/FIT?"

## 2. Retroactive Documentation
Certificate issued significantly after incident.

**CRITICAL: Compare certificate_date vs incident_date from extracted documents!**

**Thresholds:**
- 1-7 days after travel/incident: ACCEPTABLE (patient may document post-travel)
  → Set `has_minor_temporal_discrepancy=true`, `covered_reason=true`, `confidence=0.70`
- 8-14 days after: QUESTIONABLE
  → Set `covered_reason=true`, `confidence=0.65` (borderline)
- **>14 days after incident: FRAUD** (likely backdated certificate)
  → Set `is_retroactive_certificate=true`, `covered_reason=false`, `confidence=0.85`
  → Example: Incident Dec 14, Certificate Jan 5 = 22 days = FRAUD

## 3. Premature Claims
Claim filed >14 days BEFORE travel with no medical incident yet.

**NOT premature:**
- Incident already occurred → claim filed later (normal)
- Currently hospitalized, travel in future (use uncertain_prognosis instead)

**Action:** Set `is_premature_claim=true` only if filed early with no incident

## 4. Redacted Patient Information
Patient name or critical information is obscured/redacted on medical certificate.

**Indicators:**
- Black bars, white-out covering text, XXX symbols, underscores
- "Name: ___" or "Name: XXXX" or "Patient: [REDACTED]"
- Deliberately hidden information that prevents identity verification

**Action:** Set `documentation_present=false`, `covered_reason=false`, `confidence=0.90`
**Reasoning:** Cannot verify patient identity matches claimant. High-confidence denial.

## 5. Missing/Mismatched Documentation

**Requirements:**
- Medical emergency: images/PDFs with letterhead (not plain .txt/.md files)
- Theft: police report
- Description says "attached certificate" → verify document actually present

**Action:** Set `documentation_present=false` if required docs missing or wrong type

# COVERAGE RULES

**Medical Emergencies (COVERED):**
- Acute conditions: hospitalization, serious illness/injury preventing travel
- Emergency dental: abscess, severe pain requiring immediate treatment

**NOT Covered:**
- Routine checkups, preventive care, fitness certificates
- Administrative delays (appointment took too long)

**Uncertain Prognosis:**
- Patient currently hospitalized AND travel >14 days in future
- Chronic/long-term (6+ weeks recovery): `uncertain_prognosis=false`, covered
- Acute condition (unknown discharge): `uncertain_prognosis=true`, needs review

# NAME MATCHING

Compare claimant name from claim description with patient name from extracted documents.

**CRITICAL: Be VERY LENIENT with handwritten names - handwriting causes severe OCR errors!**

**ACCEPT as matching (set names_match=true):**

1. **Handwritten OCR errors (MOST IMPORTANT):**
   - Handwriting is messy and OCR makes frequent mistakes
   - Similar-looking letters get confused: o/a, e/c, n/u, i/l, B/R, y/g, v/w
   - Accept if names are "close enough" - don't require perfect match
   - Examples: "Bayante" ≈ "Benvenuti" (B similar, handwritten), "Junes" ≈ "Jones"
   - **If first name matches (even with minor OCR error), ACCEPT the claim**
   - Only reject if names are COMPLETELY different (different language/origin entirely)

2. **Cultural naming order:**
   - Italian/Hungarian/Asian surname-first: "Bonfanti Oliviero" = "Oliviero Bonfanti"
   - Spanish compound surnames: "García López María" = "María García López"

3. **Cross-language name equivalents:**
   - Oliviero (Italian) = Olivier (French) = Oliver (English)
   - Juan (Spanish) = John (English) = Jean (French)
   - María (Spanish) = Marie (French) = Mary (English)

4. **Honorifics and titles:**
   - "Dr. Smith" = "Smith" = "Dr John Smith"
   - "Mme Dubois" = "Dubois" = "Marie Dubois"

5. **Third-party claims (family):**
   - Parent filing for child (different last names acceptable)
   - Spouse filing for partner
   - Set `is_third_party_claim=true` for these cases

**REJECT as mismatch (set names_match=false):**
- ONLY if names are completely different with no possible OCR/cultural explanation
- Example: "John Smith" vs "Wang Wei" (different origin, no OCR path)

**Default stance:** When in doubt, SET `names_match=true`. Handwritten documents have poor OCR quality.

# CONFIDENCE SCORING

- 0.85-1.0: Clear evidence, definitive decision
- 0.70-0.84: Solid documentation, minor issues
- 0.60-0.69: Ambiguous but decision clear
- <0.60: Insufficient information

**Be decisive:** Documentation present + no fraud = assign 0.75+

# FLAG DEFINITIONS

- `names_match`: Claimant name matches patient name (considering cultural variations, OCR errors, language equivalents)
- `is_third_party_claim`: Filing for family member (name difference expected)
- `is_healthy_patient_certificate`: Certificate proves fitness/health, not illness (fraud pattern)
- `is_retroactive_certificate`: Certificate issued >14 days after incident (backdated fraud)
- `is_premature_claim`: Filed >14 days before travel, no incident yet (needs review)
- `has_minor_temporal_discrepancy`: Certificate 1-7 days after travel (acceptable delay)
- `uncertain_prognosis`: Currently sick, travel >14 days away, unclear if condition persists
